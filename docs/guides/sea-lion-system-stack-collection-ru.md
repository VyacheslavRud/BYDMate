# Сбор системных APK и служб BYD Sea Lion 07

Эта инструкция предназначена для BYD Sea Lion 07 2025 года, китайской версии с DiLink 5.0. Она помогает один раз получить системные APK, framework и диагностическую информацию, чтобы затем исследовать приборную панель полностью на компьютере.

Скрипт работает только на чтение. Он не устанавливает приложения, не меняет настройки автомобиля, не перезагружает DiLink, не запрашивает root и не вызывает неизвестные Binder-команды.

Актуальная версия сборщика — `v3`. Она исправляет потерю оставшихся пакетов из-за чтения `adb` из stdin, отделяет предупреждения успешных команд и progress `adb pull` от настоящих ошибок, исключает тяжёлые `compiled`-артефакты из priority archive и создаёт для него отдельный `SHA256SUMS-priority.txt`.

## Что понадобится

- MacBook с проектом BYDMate;
- автомобиль должен стоять на месте в режиме `P`;
- экран DiLink должен оставаться включённым;
- ADB должен быть включён по USB или Wi-Fi.

На этом Mac `adb` уже установлен. `jadx` и `apktool` понадобятся позднее для анализа файлов, но не нужны для самого сбора.

## Шаг 1. Включить ADB в машине

Если ADB уже использовался для установки BYDMate Dev, повторно активировать его не нужно.

В инженерном меню DiLink 5 должны быть включены один или оба переключателя:

- `Wireless adb debug switch` — для подключения по Wi-Fi;
- `Debug mode when USB is connected` — для подключения кабелем.

Иллюстрированная инструкция находится в файле [dilink5-adb-activation-ru.pdf](./dilink5-adb-activation-ru.pdf).

Не используйте стороннюю удалённую активацию через отправку QR-кода неизвестному человеку. Для сбора нужен только уже доступный режим ADB.

## Шаг 2. Подключить Mac к автомобилю

### Вариант A: USB

1. Подключите Mac к USB-порту автомобиля, который поддерживает передачу данных.
2. На экране автомобиля подтвердите доверие компьютеру, если появится запрос.
3. Откройте Terminal на Mac и выполните:

```bash
adb kill-server
adb start-server
adb devices -l
```

В списке должна появиться одна строка со статусом `device`.

### Вариант B: Wi-Fi

1. Mac и DiLink должны находиться в одной сети.
2. Посмотрите IP-адрес в инженерном меню рядом с `Wireless adb debug switch`.
3. В Terminal выполните, заменив адрес на фактический адрес машины:

```bash
adb kill-server
adb start-server
adb connect IP_МАШИНЫ:5555
adb devices -l
```

Пример IP из чужой инструкции использовать нельзя: адрес вашей машины может отличаться.

### Как выглядит правильный результат

```text
List of devices attached
192.168.1.50:5555    device product:... model:...
```

Статусы `offline` и `unauthorized` не подходят. Если открыт Android Emulator, выключите его, чтобы случайно не выбрать эмулятор вместо автомобиля. Сам скрипт дополнительно запрещает сбор с эмулятора и проверяет, что устройство идентифицируется как BYD/DiLink.

## Шаг 3. Запустить сборщик

В том же Terminal выполните:

```bash
cd /Users/vyacheslavrud/Desktop/Projects/BYDMate
./scripts/collect-byd-system-stack.sh
```

Скрипт покажет:

- ADB-адрес;
- производителя;
- модель;
- версию Android;
- fingerprint прошивки.

Убедитесь, что это автомобиль. После этого введите строго:

```text
COLLECT
```

Во время работы:

- не выключайте DiLink;
- не отсоединяйте USB или Wi-Fi;
- не запускайте Android Emulator;
- можно оставаться на любом обычном экране автомобиля;
- маршрут Waze запускать не требуется.

Часть файлов может быть закрыта SELinux. Если команда действительно завершится ошибкой, скрипт запишет отказ в `collection-errors.log` и честно пометит результат как `PARTIAL` либо `INSUFFICIENT`. Предупреждения команд, завершившихся успешно, сохраняются отдельно в `inventory-stderr.log` и не снижают статус. Не отключайте SELinux и не используйте root.

## Шаг 4. Найти результат

По умолчанию на рабочем столе появятся папка и два архива:

```text
BYD-System-Dump-YYYYMMDD_HHMMSS/
BYD-System-Dump-YYYYMMDD_HHMMSS.tar.gz
BYD-System-Dump-YYYYMMDD_HHMMSS-priority.tar.gz
```

В конце Terminal также покажет:

- статус `COMPLETE`, `PARTIAL` или `INSUFFICIENT`;
- полный путь к папке;
- размеры обоих архивов;
- SHA-256 обоих архивов.

Сначала передайте только файл `-priority.tar.gz`. В нём находятся диагностические списки, APK наиболее важных системных компонентов и собственный `SHA256SUMS-priority.txt`; каталоги `packages/*/compiled/` туда не входят. Полный `.tar.gz` дополнительно содержит framework, APEX, native-файлы, compiled-артефакты и другие большие файлы; он может занимать от сотен мегабайт до нескольких гигабайт и понадобится только по отдельному запросу.

Оба архива являются приватными: они содержат проприетарное программное обеспечение автомобиля и могут содержать названия текущих системных компонентов, процессов, дисплеев и пакетов. Не публикуйте их и не загружайте на сторонний файлообменник. Если даже приоритетный архив не помещается в чат, сначала сообщите его размер.

### Как понимать итоговый статус

- `COMPLETE` — сборщик не обнаружил ошибок выполнения или чтения;
- `PARTIAL` — основная информация получена, но некоторые операции или файлы оказались недоступны;
- `INSUFFICIENT` — не удалось получить ключевые APK, поэтому сбор нужно повторить до начала анализа.

При `PARTIAL` сначала передайте приоритетный архив вместе с текстом из Terminal. При `INSUFFICIENT` не считайте сбор законченным: откройте `collection-errors.log`, проверьте соединение ADB и повторите процедуру.

Для этой машины в `inventory/priority-package-status.txt` ожидаются пять строк `PULLED`:

```text
com.xdja.containerservice PULLED
com.byd.launchermap PULLED
com.byd.clusterdebug PULLED
com.ts.car.someip.service PULLED
com.byd.someipsystemservice PULLED
```

Если сборщик сообщает `Collector version: 1` или `2` либо выгружен только один APK, это старый результат — повторите сбор актуальным скриптом.

## Если подключено несколько устройств

Посмотрите нужный идентификатор:

```bash
adb devices -l
```

Затем запустите сборщик так:

```bash
./scripts/collect-byd-system-stack.sh --serial IP_МАШИНЫ:5555
```

При USB-подключении вместо IP будет буквенно-цифровой serial из первой колонки `adb devices -l`.

## Частые ошибки

### `no online ADB device found`

Mac не видит автомобиль. Проверьте кабель или Wi-Fi, ADB-переключатель и повторите `adb devices -l`.

### `unauthorized`

Разблокируйте экран DiLink и подтвердите доверие компьютеру. Затем выполните:

```bash
adb kill-server
adb start-server
adb devices -l
```

### `more than one online device found`

Выключите эмулятор либо используйте параметр `--serial`.

### `target does not identify itself as BYD/DiLink`

Скрипт остановился до сбора, потому что выбранное устройство не похоже на автомобиль. Снова проверьте список `adb devices -l`.

### В `collection-errors.log` есть `Permission denied`

Это возможно для части системных каталогов. Не включайте root и не пытайтесь перемонтировать разделы. Итог будет `PARTIAL` или `INSUFFICIENT`; при `PARTIAL` доступных файлов может быть достаточно для анализа.

### `output must be outside the Git working tree`

Сборщик не позволяет сохранять проприетарные файлы внутри проекта BYDMate, включая обход через `..` или символическую ссылку. Используйте путь на рабочем столе. Для собственного имени папки её родительская папка уже должна существовать:

```bash
./scripts/collect-byd-system-stack.sh --output "$HOME/Desktop/My-BYD-Dump"
```

## Что будет происходить после сбора

На Mac будут установлены инструменты анализа:

```bash
brew install jadx apktool
```

Далее исследуются:

1. `android.os.IAutoContainer` и его Binder-методы;
2. команды контейнера `16`, `18` и `0`;
3. `com.xdja.containerservice` и источник `Surface`;
4. `com.byd.launchermap` и `MyNavigationService`;
5. `com.byd.clusterdebug`;
6. SOME/IP-службы и связанные native-библиотеки;
7. разрешения, необходимые для вывода изображения на приборку.

Только после статического анализа будет подготовлен один адресный тест. Работающая навигация на лобовом стекле этим сбором не затрагивается.

Уже подтверждённые выводы и границы безопасности сохранены в [sea-lion-07-dilink5-compatibility-passport.md](./sea-lion-07-dilink5-compatibility-passport.md). Проприетарные файлы из дампа в Git не добавляются.
