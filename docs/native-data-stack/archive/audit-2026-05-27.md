## Summary

Phase 1b (native autoservice read path) завершена. D+ как читалка данных исключён — все рантайм-параметры (SOC, мощность, температуры, ёмкость, V/I, GPS gun, SoH, пробег) идут через `NativeParsReader` + autoservice Binder. D+ остаётся только как **command bus** для записи (окна, климат, свет, замки) через HTTP `localhost:8988/api/sendCmd` — этот канал держат `ActionDispatcher` (автоматизации) и `AlicePollingManager` (Smart Home).

Артефакты прошлой архитектуры частично остались в дереве: dead branch у `TrackingService`, watchdog файлы, orphan strings в `values/strings.xml`, упоминания toggle и ADB-инструкций в README. Архитектурных рисков нет, но требуется cleanup и доработки adaptive loop / batch reads / non-Leopard3 поддержки.

## 1. UI Toggle на скриншоте v2.8.1 — Stale APK Verdict

**Жалоба пользователя:** на скриншоте Settings v2.8.1 виден переключатель источника данных (energydata / DiPlus TripInfo). В коде HEAD его быть не должно.

**Вердикт:** установленный APK собран ДО коммита удаления toggle. Это не баг кода.

### Доказательства

1. Коммит удаления: `3d3f7d5 refactor(nativestack): drop DataSource toggle from Settings UI` — 67 строк удалено из `app/src/main/kotlin/com/bydmate/app/ui/settings/SettingsScreen.kt`.
2. На HEAD enum `DataSource` имеет ровно одно значение `ENERGYDATA`. Никакой Kotlin строкой UI не рендерится:
   ```
   git grep -n "settings_trips_datasource_header\|settings_datasource_switching\|settings_datasource_done" -- 'app/src/main/kotlin/**/*.kt'
   → 0 matches
   ```
3. `SettingsScreen.kt:434-468` — функция `TripsSection()` рисует только autoservice toggle, без выбора источника.

### Orphan strings (остались в xml, не читаются кодом)

`app/src/main/res/values/strings.xml:42-44` и `app/src/main/res/values-en/strings.xml:31-33`:
- `settings_trips_datasource_header`
- `settings_trips_datasource_note`
- `settings_trips_datasource_hint`
- `settings_datasource_switching`
- `settings_datasource_done`
- `welcome_model_step_description`

Все шесть строк подлежат удалению в Этапе A — они уже мёртвый код в ресурсах.

### Почему `versionName` показывает "2.8.1"

`lastUpdateTime` в `dumpsys package com.bydmate.app` обновляется при push APK в `/sdcard/Download`, даже если фактическая переустановка поверх не произошла. Сам по себе этот атрибут не доказывает, что новый APK дошёл до dex-runtime — это лишь время последнего касания файла. Версия "2.8.1" в UI могла быть зашита в манифест уже несколько сборок назад, при том что dex-классы остались от предыдущей.

**Урок:** при противоречии "UI vs код" обязательно сравнить sha256 установленного APK с локально собранным, и проверить git log на удаление UI элемента — без proof не объявлять "stale install" (см. memory `feedback_apk_freshness_check.md`).

## 2. D+ Inventory — что осталось

### 2.1. Write bus (hard dependencies — НЕ удалять до Phase 2)

| Файл | Использует | Зачем |
|------|-----------|-------|
| `app/src/main/kotlin/com/bydmate/app/data/diplus/DiParsControlClient.kt` | — | HTTP POST на `127.0.0.1:8988/api/sendCmd`. Единственный write bus в проекте. |
| `app/src/main/kotlin/com/bydmate/app/data/automation/ActionDispatcher.kt:24,84` | `DiParsControlClient` | Выполнение D+ команд из правил автоматизации (окна/климат/свет). |
| `app/src/main/kotlin/com/bydmate/app/data/smarthome/AlicePollingManager.kt:25,113` | `DiParsControlClient` | Полученные с VPS Yandex Smart Home команды отправляются на машину тем же HTTP. |

### 2.2. Dead branch (удалить в Этапе A)

| Файл / location | Что | Почему dead |
|-----------------|-----|-------------|
| `DiPlusWatchdog.kt` | Watchdog объект, мониторил здоровье D+ когда он был read source | Read source = autoservice. Watchdog не наблюдает за write bus, не нужен. |
| `TrackingService.tryLaunchDiPlus()` (lines 903-951) | Попытка перезапустить D+ через `AdbOnDeviceClient` | Use case "перезапустить D+ потому что read залип" исчез вместе с read path. |
| `TrackingService` — поля `lastDiPlusRelaunchAt`, `diPlusRelaunchCount` и связанная state | Учёт relaunch-цикла | Без `tryLaunchDiPlus()` поля сирые. |
| `AdbOnDeviceClient.launchDiPlusService()` + `LAUNCH_DIPLUS_CMD` константа | shell-uid `am start-foreground-service` для D+ | Не вызывается ни из одного активного code path после удаления `tryLaunchDiPlus()`. |
| Orphan strings (см. секцию 1) | xml ресурсы | UI элемент удалён, строки висят. |

**Эмпирическое подтверждение:** юзер выключил D+ UI на машине, окна/климат/Smart Home/Automation продолжали работать. Гипотеза "D+ — единственный command bus" опровергнута. Возможные объяснения:

1. `aps_diplus` daemon (см. memory `reference_aps_diplus_daemon.md`) живёт независимо от UI приложения D+ до перезагрузки.
2. Foreground service `com.van.diplus/service.MainService` остался активен в фоне после выключения UI.
3. HTTP-сервер на `localhost:8988` мог быть hostnut другим компонентом (этот пункт надо валидировать перед удалением command bus).

**Это нужно подтвердить эмпирически** перед Phase 2 — например `adb shell ps -A | grep diplus` после полного kill D+. Без этого нельзя удалять `DiParsControlClient`.

### 2.3. UI naming / диагностика (касается, но не критично)

- `SettingsScreen.kt` — упоминания "DiPlus" в подписях экспериментальных переключателей могут переименоваться на "autoservice" / "native data" для честности (D+ больше не read source).
- `LogScreen` / `DiagnosticsScreen` — фильтры по тегу `DiPlus*` остались. Полезны пока живёт write bus, но названия групп могут запутывать.

## 3. Архитектурные находки

### 3.1. NativeParsReader — sequential 40 fids на каждый poll

`app/src/main/kotlin/com/bydmate/app/data/nativestack/NativeParsReader.kt:28-49` исполняет 40 `adb shell` транзакций последовательно. При `TrackingService.POLL_INTERVAL_MS = 3000L` (3 секунды) уже p95 риск задержек на старте, когда autoservice холодный.

**План:** Этап B Phase 1.5 — batch + async + adaptive cadence в shared adaptive loop. Возможно, перевод части fids на `Parcel.writeIntArray` / multiplexed transact.

### 3.2. ABRP cadence не соответствует политике

`IternioIntervalPolicy.kt` декларирует DRIVING=1 секунда, но фактический pull идёт из `TrackingService.POLL_INTERVAL_MS = 3000L` — ABRP получает 0.33 Hz вместо 1 Hz.

**Решение пользователя:** не делать отдельный telemetry loop. Один shared 1Hz adaptive loop = single source of truth для ABRP, виджета, дашборда. Реализуется в Этапе B Phase 1.5.

### 3.3. TripTracker не пишет TripEntity

`app/src/main/kotlin/com/bydmate/app/data/tracking/TripTracker.kt` — GPS-only collector. Все `insertTrip` живут внутри `HistoryImporter.kt:167` и `:268` (`syncFromEnergyData()`).

**Следствие:** на машинах без BYD energydata SQLite (Song, Atto, любые не-Leopard3 модели) **не запишется ни одной поездки**. GPS-точки накопятся в `TripGpsPoint` с `tripId=0` и осиротеют.

**План:** Этап B Phase 1.5 — native trip recording из shared adaptive loop поверх autoservice SOC + mileage + speed. Универсализация архитектуры на all-BYD.

### 3.4. TrackingService — god class

`TrackingService.kt` тащит на себе: GPS, ADB lifecycle, D+ relaunch (dead), Native poll, AI insights trigger, autostart logic, sync trigger. Кандидат на декомпозицию после стабилизации native stack.

### 3.5. BYDMateApp.runBlocking

`BYDMateApp.onCreate()` использует `runBlocking` на старте — блокирует main thread на инициализацию ADB / autoservice connection. На холодном старте DiLink (когда autoservice ещё не поднят) даёт визуальный freeze в первые 1-3 секунды.

## 4. Что подтверждено OK

- **D+ read path удалён.** `DiPlusReader` / `DiPlusDbReader` больше не дёргаются для live данных. Только `DiPlusDbReader` для исторического TripInfo enrichment остаётся в `HistoryImporter`, и это OK — это импорт прошлых поездок, не runtime.
- **autoservice write channel заблокирован.** Direct tx=6 даёт `SecurityException` на L3 — все попытки записи через autoservice отбиваются на уровне system service (см. memory `feedback_seat_heat_unavailable_leopard3.md`).
- **Release не debuggable.** Подтверждено по APK manifest — `android:debuggable="false"`. Соответствует требованию из memory `reference_diplus_backend_x2x_fun.md` (защита от D+ APK exfiltration).
- **Sentinel containment / liveness gating / snapshot pattern реализованы.** Чтения с sentinel классами фильтруются (см. memory `reference_autoservice_sentinel_classes.md`). SoH `1145045032` валидирован на S9.
- **Signing OK.** Release собран и подписан `bydmate-release.jks` (alias bydmate), путь `~/.android/keystores/bydmate-release.jks`.

## 5. Removal Plan

### Этап A — D+ dead code cleanup (in progress, 2026-05-27)

Цели: убрать всё, что относится только к D+ как read source. Write bus не трогать.

Удалить:
- `DiPlusWatchdog.kt` (файл целиком)
- `TrackingService.tryLaunchDiPlus()` (lines 903-951) + поля `lastDiPlusRelaunchAt`, `diPlusRelaunchCount`, любая state вокруг relaunch
- `AdbOnDeviceClient.launchDiPlusService()` + `LAUNCH_DIPLUS_CMD` константа
- Orphan strings в `values/strings.xml` и `values-en/strings.xml` (6 ключей, см. секцию 1)
- README блок "Источник данных поездок" (около `README.md:215`)
- README блок "ADB активация" с пользовательскими ADB-инструкциями (около `README.md:308`) — нарушает `feedback_permissions_standard_ui`
- Возможно переименовать UI ярлыки "DiPlus" → "native data" в Settings

### Этап B Phase 1.5 — shared adaptive loop (planned)

Цели: один источник truth, batch + async, native trip recording.

Сделать:
- Shared adaptive 1Hz vehicle loop (driving) / 5s (idle) / sleep-mode handling
- Batch reads для NativeParsReader (multiplexed transact или parcel array)
- Native trip recording из loop поверх SOC + mileage + speed → закрывает gap для не-Leopard3 (см. `feedback_trip_recording_gap_non_leopard3.md`)
- ABRP cadence теперь матчит политику Iternio (1 Hz driving)
- Возможный refactor `TrackingService` god-class

### Этап C Phase 2 — write channel migration (planned)

Цели: убрать зависимость от D+ полностью.

Опции:
1. **DataStream bridge к shell-uid daemon** (по аналогии с `aps_diplus`, см. memory `reference_autoservice_write_channel.md`). Технически воспроизводимо.
2. **Cloud impersonation** — закрыт исследованием (см. `reference_cloud_impersonation_closed.md`).
3. **Signature permission через подпись** — нереалистично без BYD ключа.

После Phase 2:
- Удалить `DiParsControlClient.kt`
- Заменить вызовы в `ActionDispatcher` и `AlicePollingManager` на native write bus
- D+ становится не нужен ни как read, ни как write — пользователь может удалить D+ с DiLink
