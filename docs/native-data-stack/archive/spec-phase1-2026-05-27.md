# Native Data Stack ‚Äî Phase 1 Design Spec

**Date:** 2026-05-27
**Branch:** `feature/native-data-stack`
**Status:** Draft for user review
**Phase:** 1 of 3 (read migration). Phase 2 = write, Phase 3 = cleanup.

## 1. Goal & Scope

Remove BYDMate's dependency on the third-party DiPlus (D+) app for read data, without changing the current end-user feature set.

### In scope

- 47 live parameters currently read via `DiParsClient` (HTTP `localhost:8988/api/getDiPars`) ‚Üí read via `autoservice` Binder (`AdbOnDeviceClient` ‚Üí ServiceManager) using validated feature-ids (fids).
- `DiPlusDbReader` (SQLite `van_bm_db.TripInfo`) ‚Äî deprecated and unused in Phase 1b; actual file removal in Phase 3. SOC and avg-speed enrichment moves to `FID_SOC` realtime + `TrackingService` captured values.
- `HistoryImporter.syncFromDiPlus()` (line 455) ‚Äî removed in Phase 1b along with the `DataSource.DIPLUS` setting and its UI toggle in `SettingsScreen.kt` and `WelcomeScreen.kt`. Trip import is energydata-only post Phase 1b.
- `AutoserviceChargingDetector` D+ fallback paths (`diParsClient.fetch()` at lines 141 and 314, for SOC/cell/temp) ‚Äî removed. Native fids cover all three.
- ChargingLog: `DiPlusDbReader` exposes only `readTripInfo()`; no ChargingLog accessor exists in BYDMate code. Native charging session detection via `AutoserviceChargingDetector` is authoritative.

### Out of scope (separate phases)

- Phase 2: replace `DiParsControlClient.sendCommand()` (write) with `autoservice` setInt by whitelist.
- Phase 3: delete D+ modules, watchdog, manifest entries.
- New UX features from external feature inventory (voice control, hardware button binding, FAB quick-actions, Battery V/I dashboard, cycle stats).

### Success criteria

- Blocking groups 1-4 (see ¬ß7.3): every parameter populated via `autoservice` with values matching D+ within tolerance across documented vehicle states.
- Desirable groups 5-6: same coverage, or documented gap in `docs/native-data-stack/gaps.md`.
- Nice-to-have group 7: best effort.
- BYDMate runs identically with D+ disabled or uninstalled.
- `docs/native-data-stack/fid-map.md` ‚Äî public fid catalog for Leopard 3.
- Zero references to `DiParsClient.fetch()` or `DiPlusDbReader` remain in production code paths (only the classes themselves, pending Phase 3 removal). No D+ fallback anywhere.

## 2. Architecture ‚Äî Phase 1a (Validation Suite)

Standalone Mac-side toolchain. No changes to production app code.

### Components

### fid-candidates.yaml schema


### Diff outcomes per fid

| Outcome | Condition | Action |
|---|---|---|
| PASS | autoservice value matches D+ within tolerance across N‚â•5 snapshots in ‚â•2 states | Mark `validated`, include in `FidMap` |
| TOLERANCE | Semantic match but scale differs (raw int vs scaled, signed vs unsigned) | Write decoder in `ParamDecoder.kt`, re-snap |
| SENTINEL | autoservice returns sentinel from `SentinelDecoder` (0xFFFFD8E5, -10011, etc) | Fid not available on Leopard 3 ‚Äî find alternative in catalog or via logcat reverse. Phase 1b not merged until resolved. No D+ fallback. |
| MISSING | D+ has value, no autoservice candidate identified | Open question, separate research |
| MISMATCH | Values differ, not sentinel, not scaling | Investigate. Cross-check with energydata if applicable. Do not include. |

### Tolerance per data type

- Int % (SOC, fan level, window position %) ‚Äî exact
- Float kW (engine power, charging power) ‚Äî ¬±0.5 kW
- Float V (12V aux, cell voltages) ‚Äî ¬±0.01 V
- Int ¬∞C (cabin, battery, exterior temp) ‚Äî ¬±1¬∞C
- Int kPa (tyre pressure) ‚Äî exact
- Enum int (gear, lock, gun, work_mode) ‚Äî exact + mapping table in decoder

### Vehicle states

All snapshots taken with IGN ON (DiLink awake, ADB reachable).

| State | Setup | Covers |
|---|---|---|
| S2: parked + IGN ON | gear=P, AC off | baseline: doors/windows/locks/cabin/12V/SOC |
| S3: driving | gear=D, 5-10 min run | live ENG_POW, speed, mileage delta, motor temp |
| S4: AC active | climate ON, fan ‚â•2, target temp set | AC params, insideTemp response |
| S5: AC charging | gun connected, charger online | charging_state, gun_state, capacity counter, charging_power |
| S6: sleep-mode probe | IGN OFF, gun in, charging ‚Äî DiLink screen off | does autoservice respond? Decides if background polling is viable. Does NOT block Phase 1b. |

Minimum: 5 snapshots per fid per applicable state. Increase to 10 if variance is high.

### Operator loop

1. Operator (user) takes car to required state, confirms readiness.
2. Engineer (assistant) runs ADB snapshot batch from Mac.
3. Engineer reports per-param outcomes (PASS / TOLERANCE / SENTINEL / etc).
4. Operator confirms readings reflect physical reality (e.g. window 30% open vs reported value).
5. Operator advances to next state; repeat.
6. Full report written; public fid-map updated.

### Safety net during validation

- Transact 5 (getInt) and 7 (getFloat) for scalar fids. Transact 9 (getBuffer) is also read-only and allowed where a fid returns an array (existing `AdbOnDeviceClient` whitelist `5|7|9` is correct; tx=6 setInt remains hard-forbidden until Phase 2).
- Pre-session check: `adb shell service check autoservice` returns `found`.
- ‚â§5 autoservice req/sec to avoid Binder rate-limit.
- Stop immediately on sentinel/mismatch for safety-critical fids (SOC, charging_state, gun_state).

### Validation criteria for derived counters

Standard PASS criteria are insufficient for stateful counters like `totalElecConsumption`. For these fields, additionally require:

- **Monotonicity**: native counter never decreases within a contiguous IGN-on session.
- **Step size sanity**: per-snapshot delta plausible against `FID_ENGINE_POWER` integral over the interval.
- **IGN-off continuity**: value resumes at or above the prior IGN-off reading.
- **p95 read latency ‚â§ 500ms** across S2-S5 (per-fid), to ensure UI polling at 2s interval is not starved.
- **Null continuity**: fid does not flap between sentinel and valid within a single state.

## 3. Architecture ‚Äî Phase 1b (Code Migration)

Production code change. Depends on Phase 1a completion (validated fid-map.md).

### New module

```
app/src/main/kotlin/com/bydmate/app/data/nativestack/
‚îú‚îÄ‚îÄ NativeParsReader.kt   # implements ParsReader, replaces DiParsClient.fetch()
‚îú‚îÄ‚îÄ NativeChargingReader.kt # optional, only if ChargingLog migration needed
‚îú‚îÄ‚îÄ ParamDecoder.kt       # raw int/float ‚Üí typed values
‚îî‚îÄ‚îÄ FidMap.kt             # constant map: DiParsData field ‚Üí (device, fid, tx, decoder)
```

### Contract preservation

`NativeParsReader.fetch(): DiParsData?` is shape-identical to `DiParsClient.fetch()`. The interface change requires updating every direct `DiParsClient` injection to `ParsReader`. Confirmed direct-injection sites (must be updated in Phase 1b):

- `TrackingService.kt:60` ‚Äî main polling loop at `TrackingService.kt:636` consumes `DiParsData` ~every 2s; this is the hottest path.
- `AutoserviceChargingDetector.kt:57` ‚Äî currently uses D+ as fallback for SOC/cell/temp; fallback removed (see In Scope).
- `SettingsViewModel.kt:140` ‚Äî used for diagnostics; log tags must be renamed accordingly.
- `HistoryImporter.kt` ‚Äî `enrichWithDiPlus()` becomes a no-op stub. SOC and `avgSpeedKmh` for retroactive enrichment now sourced from `TrackingService` captured values (already collected for live tracking; extend to import path).
- `AlicePollingManager.kt` ‚Äî read path only; write path stays D+ until Phase 2.
- `BatteryHealthDialog`, `IternioTelemetryClient`, `DashboardScreen`, `GunStateEdgeDetector`, `ActionDispatcher` (trigger conditions) ‚Äî verify no direct `DiParsClient` import remains; all access through `ParsReader`.

A repository-wide `rg "DiParsClient"` check must pass with zero hits in production sources after Phase 1b (excluding the class definition itself, which is removed in Phase 3).

### ParsReader interface

```kotlin
interface ParsReader {
    suspend fun fetch(): DiParsData?
}
```

`DiParsClient` already matches this shape. Both implementations co-exist briefly; production binding swaps to `NativeParsReader` in `AppModule`. Old `DiParsClient` is **kept in the codebase only until Phase 2 ships**, after which Phase 3 removes it. No runtime feature flag ‚Äî single binding compiled in.

### Hilt swap (AppModule)

Replace:

```kotlin
@Provides @Singleton
fun provideDiParsClient(httpClient: OkHttpClient): DiParsClient = DiParsClient(httpClient)
```

With:

```kotlin
@Provides @Singleton
fun provideParsReader(
    autoservice: AutoserviceClient,
    fidMap: FidMap
): ParsReader = NativeParsReader(autoservice, fidMap)
```

Inject `ParsReader` (interface) wherever `DiParsClient` was injected.

### Computed / aggregated fields (not raw BMS)

A subset of `DiParsData` fields are derived inside D+ rather than direct raw BMS reads. These require a semantic-equivalent native source, not a fid copy:

| DiParsData field | DiPlus TEMPLATE token | Semantic | Native source candidate |
|---|---|---|---|
| `totalElecConsumption` | `{ÊÄªÁîµËÄó}` | Cumulative lifetime kWh throughput | `FID_LIFETIME_KWH` (1032871984, dev=1014, tx=7) ‚Äî already in FidRegistry. Phase 1a must validate monotonicity and value equivalence to D+ output across IGN cycles. |
| `batteryCapacityKwh` | `{ÁîµÊ±ÝÂÆπÈáè}` | Battery pack nominal capacity (kWh) | Static config from `SettingsRepository.batteryCapacityKwh` (user-set). D+ likely reports a hard-coded model constant; native source is the existing user setting, not a fid. Verify in Phase 1a snap whether D+ value is constant or changes with conditions. |

`TrackingService` consumes `totalElecConsumption` in the polling loop (`TrackingService.kt:359, 368, 402, 694, 700, 705, 726, 744`) for session start/end, SOC interpolation, range estimation. Migration must produce identical values to D+ to avoid regression in trip records and range UI.

### TripInfo (D+ SQLite)

`DiPlusDbReader` becomes deprecated and unused in Phase 1b. `HistoryImporter.enrichWithDiPlus()` becomes a no-op stub. SOC for trip start/end is taken from `FID_SOC` realtime captured by `TrackingService` at trip start and end (already happens for live tracking; extend to import path).

Native energydata trip source (`EnergyDataReader`) is unchanged. It remains the primary trip source.

### ChargingLog

`DiPlusDbReader.readChargingLog()` is empty on observed Leopard 3 DBs and there is already a working native detector (`AutoserviceChargingDetector` + `GunStateEdgeDetector`). No migration needed. Calibration option below.

### Charging detection calibration

After Phase 1a snap of charging fids, decide:

- **Keep current logic**: gun-edge primary trigger, capacity counter for kWh.
- **Calibrate**: use `charging_state` (fid candidate 1231032336, dev=1009) as authoritative session trigger, `charger_work_state` (fid candidate 666894346) to filter "gun connected but charger offline" cases, `charge_stop_soc` (fid candidate 1132462120) to display target SoC, `charging_power_dd` (fid candidate 842006552, float kW) for power profile.

Decision made on Phase 1a data, not pre-committed. If new fids produce identical session boundaries as current edge-detector, current logic is kept and only data source migrates.

### Refresh policy (event-driven model)

An event-driven refresh pattern is feasible: only `vehicle_state` and `instrument_ok_indicator` are polled periodically; ~23 secondary fids are re-read on master-signal change. Whether to adopt this pattern in BYDMate is **a Phase 1b implementation decision** based on Phase 1a latency benchmarks. If current per-poll latency is acceptable, keep simple periodic poll.

### Module changes outside data/nativestack/

- `AppModule.kt` ‚Äî Hilt binding swap.
- `HistoryImporter.kt` ‚Äî remove DiPlus enrichment branch, capture SOC from `TrackingService`-provided start/end values.
- `AlicePollingManager.kt` ‚Äî read path uses `ParsReader`; write path stays D+ until Phase 2.

### Modules untouched in Phase 1b

- `DiParsControlClient` ‚Äî Phase 2.
- `DiPlusWatchdog` ‚Äî needed while control still goes through D+. Phase 3 removes it.

## 4. Testing

### Phase 1a outputs feed Phase 1b tests

Each validated snapshot pair (D+ raw, autoservice raw) becomes a regression fixture:

```
app/src/test/resources/native-stack-fixtures/
‚îú‚îÄ‚îÄ s2-parked-on.json
‚îú‚îÄ‚îÄ s3-driving.json
‚îú‚îÄ‚îÄ s4-ac-active.json
‚îî‚îÄ‚îÄ s5-charging.json
```

### Unit tests

- `NativeParsReaderTest` ‚Äî for each fixture: feed autoservice raw bytes ‚Üí assert `DiParsData` field-by-field equals `DiParsClient.parse(diplus_raw)` for the same snapshot.
- `ParamDecoderTest` ‚Äî encode/decode edge cases per data type (sentinel values, negative kW, percentages).
- `FidMapTest` ‚Äî invariants: every `DiParsData` field has a mapping, no duplicate fid+device pairs.

### Snapshot test

Single integration test: take fixture, run through full `NativeParsReader.fetch()` (mocked `AutoserviceClient` returns raw bytes from fixture), compare result to `DiParsClient.parse(fixture.diplus_raw)`. Catches any decoder regression.

### Manual smoke (before merge)

- Main dashboard renders all cards correctly with D+ disabled (settings ‚Üí disable autostart of D+).
- Smart Home VPS state sync receives correct device states.
- Automation rules trigger on correct conditions (gear, climate, place geofence).
- Trip import produces equivalent `TripEntity` records compared to prior week's data.

## 5. Risks & Mitigation

| Risk | Severity | Mitigation |
|---|---|---|
| Fid returns sentinel on Leopard 3 with no alternative in catalog | High | Phase 1a catches this before code changes. Re-probe via logcat or alternative reference lookup. Phase 1b does not merge until resolved. |
| autoservice rate-limit or Binder DoS under batched reads | Medium | ‚â§5 req/sec, snapshot interval 2s. Benchmark latency in Phase 1a; if too slow, adopt event-driven refresh pattern. |
| `NativeParsReader.fetch()` slower than `DiParsClient.fetch()` causing UI lag | Medium | p50/p95 latency measurement in Phase 1a. Mitigate via batching or selective event-driven refresh. |
| Decoder regression changes consumer behaviour subtly | Medium | Fixture-based snapshot tests + manual smoke. |
| `HistoryImporter` SOC enrichment regression breaks trip stats | Medium | Snapshot existing trip records pre-merge; assert new flow produces same `socStart/socEnd/avgSpeed` for back-fill set. |
| `AdbOnDeviceClient` connection drop more frequent than D+ HTTP localhost | Low | Existing retry logic. Measure MTBF in Phase 1a. |
| Multi-process contention: foreground UI, background `TrackingService`, and validation scripts all share the same on-device ADB socket. `AdbOnDeviceClient.kt:68` has no cross-process lock. | Medium | Phase 1a validation must run with the app stopped (`adb shell am force-stop com.bydmate.app`). Phase 1b: instrument `AdbOnDeviceClient` with a shared mutex if benchmark shows contention; otherwise accept (single-process app in steady state). |
