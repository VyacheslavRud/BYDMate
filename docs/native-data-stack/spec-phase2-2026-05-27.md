# Phase 2 — Single Vehicle API, remove D+ dependency

**Status:** draft v2 (rewritten 2026-05-27 after operator alignment)
**Branch:** `feature/native-data-stack`
**Phase 1 spec:** `docs/native-data-stack/spec-phase1-2026-05-27.md`

---

## 1. Goal

Replace every interaction with the external D+ companion app with a single in-house
**`VehicleApi`** layer that reads and writes vehicle state via native autoservice Binder.
After Phase 2 merges, the operator can uninstall D+ and BYDMate continues to function
identically — no missing data, no broken controls, no error banners.

## 2. Why

D+ is an external companion app that exposes an HTTP proxy on `127.0.0.1:8988`. It is
the only thing between BYDMate and the vehicle today. Owning that boundary ourselves
gives us:

- Stability — D+ updates cannot break us.
- Reach — no required D+ install per user.
- Cleanliness — one client class instead of three (DiPlusClient + DiPlusDbReader + DiParsControlClient).
- Future control extensions without waiting on D+'s command surface.

## 3. Scope

### 3.1 Read parameters (146)

Every parameter the app currently consumes from D+ plus the validated additions from
Phase 1a. Source-of-truth list is the YAML at `scripts/native-stack/fid-candidates.yaml`
(146 imported entries after S8 batch import). Categories covered:

- Battery: SOC, SOH, voltage, current, cell min/max V, max charge power, EV range,
  capacity, max/min cell temp, BMS temp.
- Powertrain: total mileage, total avg speed, engine power kW, front/rear motor
  current/temp/RPM.
- Climate: AC status, modes, set/inside/outside temp, fan level, recirc, defrost
  front/rear.
- Seats: heat level (0-5) + vent level (0-5) for driver and passenger.
- Body: window FL/FR/RL/RR position %, sunroof position, sunshade position, trunk,
  hood, all four lock states.
- Lights: low/high beam, DRL, ambient, fog front/rear, interior.
- Safety: tire pressures × 4, belts × 5, alarm state, gear, brake/handbrake.
- Charging: gun state, charging type (AC/DC), charging power live kW, charge stop SoC.
- Environment: GPS lat/lon/alt/heading, rain sensor, ambient brightness, slope.
- Status: speed, odometer, drive mode (read-only).

Sentinel filtering follows `reference_autoservice_sentinel_classes` (memory): −10011
permanent drop, −10013 transient, 65535 trim-absent, −1.0 GPS no-fix.

### 3.2 Write actions (~97)

Three layers, all routed through `VehicleApi.write*()`:

**Layer A — current UI parity (55 actions):** every command currently sent via
`DiParsControlClient.sendCommand(...)`. Source: union of
`AutomationViewModel.ACTION_COMMANDS` (49) and the Smart Home VPS catalog (rear-LR
windows × 4 + trunk × 2 added by Alice). All 55 stay as the canonical baseline —
no UI change, same Chinese command strings on input, native dispatcher under the hood.

**Layer B — Leopard 3 obvious extensions (~15 actions):** physically present on
operator's vehicle, missing from current UI surface but trivial to add once write
channel works.

- Defrost rear on/off
- Seat heat levels 3-5 (driver + passenger) — current UI exposes only 1/2
- Seat vent levels 3-5 (driver + passenger)
- AC dual mode toggle
- Set driver temp / set passenger temp (separate when dual is on)
- AC fan level (continuous, 1-7)
- AC wind mode (face / feet / windshield / mixed)
- AC ctrl mode (auto / manual)
- DRL auto mode
- Sunshade position %  (currently only open/close)
- Sunroof tilt mode (rear-up "smoker" mode, operator-confirmed present)

**Layer C — other BYD models (≈27–35 actions, exact count finalized at 2d mapping):**
physically absent on Leopard 3 but present on other BYD models. Counts depend on
whether action variants (e.g. each massage program × each level) collapse to a single
`(fid, value)` family or stay as separate `action_name`s. Implemented in `VehicleApi`
for forward portability — the app will simply not surface them in UI for vehicles
where the readback returns "not supported" sentinel, but the dispatch code is there.

- Steering wheel heat on / off / lvl2 / lvl3
- Driver massage: shoulder / waist / stretch / wave / pulse + levels 1-3 + off
- Passenger massage: same shape
- Window per-side: half / stop / vent (in addition to existing open/close) — driver,
  passenger, rear-left, rear-right
- Sunroof comfort mode (slow auto open/close)

Action names use stable English identifiers (e.g. `set_driver_temp`, `driver_seat_heat_lvl3`)
matching the reference catalog. The dispatcher maps `(action_name, args)` →
`(fid, dev, value)` from a single allowlist table.

### 3.3 Hard out-of-scope (NEVER write)

Enforced at the allowlist layer. Any attempt rejected before reaching the Binder.

- Drive controls: brake, throttle, gear, **steering angle / steering assist / steering
  control writes** (note: steering wheel **heater** in Layer C is cabin comfort, not
  steering — allowed), EPB, ADAS configs.
- Firmware: anything writing to `/system`, BMS, MCU, CAN bus, gateway, OTA.
- Safety: airbag arming, ABS thresholds, seatbelt enforcement, alarm state.
- Powertrain modes: drive mode Eco/Standard/Sport (read-only telemetry).
- Service intervals / odometer (resale records).

## 4. Architecture

### 4.1 The `VehicleApi` class

One injectable class. Two method groups.

```
class VehicleApi {
    // Reads — ~146 methods. Each returns the decoded value or null on sentinel.
    suspend fun readSoc(): Float?
    suspend fun readSpeed(): Float?
    suspend fun readWindowDriver(): Int?           // 0-100 %
    suspend fun readChargeGun(): ChargeGunState?
    // ... etc

    // Writes — ~97 methods. Each returns Boolean (readback OK / fail).
    suspend fun writeAcOn(): Boolean
    suspend fun writeAcOff(): Boolean
    suspend fun writeSetDriverTemp(celsius: Int): Boolean      // 16-32
    suspend fun writeSeatHeatDriver(level: Int): Boolean       // 0-5
    suspend fun writeWindowDriver(percent: Int): Boolean       // 0/50/100
    // ... etc
}
```

Reads go straight through the existing `AutoserviceClient` (Phase 1 stack).
Writes go through a new internal helper that talks to the local daemon (§4.2).

### 4.2 Write path — small in-vehicle helper process

`service call autoservice 6 ...` (the write opcode) from our ADB-on-device path
returns `SecurityException` because autoservice rejects the calling uid for writes.
Reads (opcodes 5 and 7) pass; writes do not.

The workaround used by every other native control implementation on these head units:
start a tiny long-lived process inside the vehicle via `app_process` and have it
issue writes through the direct Java Binder interface (not the `service call`
command-line tool). For platform-internal reasons that are out of scope here, autoservice
accepts that path under the same uid.

Lifecycle:
- On app boot, BYDMate uses its existing `AdbOnDeviceClient` to push a small Kotlin
  bootstrap script and start the helper.
- Helper listens on `127.0.0.1:<port>`. BYDMate connects and sends
  `{fid, dev, value}` per write. Helper performs the Binder transact and returns
  status + readback.
- **Idempotent bootstrap.** Before spawning, BYDMate first attempts a `connect()` to
  the helper port. If it responds healthy, reuse it (no second spawn). This prevents
  multi-instance races at boot or after WorkManager retries.
- **Single-owner port.** Helper binds the loopback port exclusively; a second helper
  spawn would fail-bind and exit cleanly, leaving the first owner intact.
- If the helper dies (process restart, OS reboot, no heartbeat for N seconds) BYDMate
  re-spawns it via the same ADB path on next boot or next write.

**Critical unknown:** this approach works on similar head-unit platforms in
implementations we have seen; on Leopard 3 it is a hypothesis. Phase 2b validates it
before any code lands. If the hypothesis fails on Leopard 3, Phase 2 is unreachable
through autoservice and we stop and re-plan. This is the single biggest risk.

### 4.3 ADB-on-device bootstrap API (narrow)

The current `AdbOnDeviceClient.exec()` allows only `service call autoservice {5,7,9}`
for read safety (Phase 1 constraint). For the bootstrap step in §4.2 we add **two
specifically-scoped methods**, NOT a generalized push/spawn:

- `bootstrapHelper()`: pushes a **fixed asset** from the app's `assets/` to a **fixed
  private destination path** (`/data/local/tmp/<fixed-name>`), then runs a **hardcoded
  command line** with no caller-provided arguments. SHA-256 of the asset is compiled
  into the app; the method refuses to spawn if the on-device file does not match.
- `helperHeartbeat()`: read-only `ps` check (or equivalent) for the known process
  name. No spawn, no kill.

Generic file-push or generic command execution are **not exposed** — only these two
fixed operations. Read-side calls retain their existing `service call autoservice
{5,7,9}` restriction.

## 5. Safety

Three rules, applied in `VehicleApi.write*()` before any Binder call.

1. **Allowlist.** Code-hardcoded map `(action_name) → (fid, dev, value-range, category)`.
   Unknown action name = rejected. Category is informational (used in audit log),
   not gate — the per-action entry IS the gate.

2. **Readback.** After every successful Binder write, immediately read the
   corresponding state fid. If readback ≠ expected, return false and log "readback
   mismatch". UI shows error to the operator.

3. **Audit log.** Local Room table `vehicle_write_log` (timestamp, action_name, fid,
   dev, requested, readback, status). Last 200 entries shown in Settings →
   Diagnostics. Local only — never sent to VPS / telemetry.

No speed gate. No first-write modal. No biometric. No category-level enforcement. No
undo. No batch protection. Operators know when it is safe to open a window.

**Smoke checklist (end of 2e, before 2f):**

1. App launches without errors after D+ stays uninstalled.
2. Dashboard live data updates while driving (SOC ticks, speed shown, AC state correct).
3. One trip writes a complete row to Room + GPS points attached.
4. One charging session writes a complete row to Room with start/end SOC.
5. One automation rule fires its action successfully (e.g. open driver window).
6. One Yandex Alice voice command executes successfully end-to-end.
7. ABRP receives a telemetry packet during driving (visible in ABRP dashboard).
8. Settings → Diagnostics panel shows live values from `VehicleApi`.

Operator runs this checklist once. No 97-item regression matrix — the features are
already proven, only their data source changes.

## 7. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Helper hypothesis fails on Leopard 3 in 2b | CRITICAL | Stop, re-plan. Phase 2 is unreachable via autoservice writes without it. Document the failure mode for future paths. |
| Write goes to wrong (fid, dev) namespace and silently corrupts an unrelated parameter — read fids and write fids live in different dev spaces, accidental crossover is plausible | HIGH | 2b's first write is a **no-op** (write the parameter's current value back to itself). Always read an independent state fid for readback, not the same address that was written. Stop the migration on any unexpected mismatch. Audit log captures every (action, fid, dev, requested, readback) for forensic review. |
| A fid from the reference catalog returns wrong / no data on Leopard 3 | MEDIUM | Found at 2c probe time. Report to operator, drop that single action from Layer A/B, keep it stubbed in Layer C for other models. Migration not blocked. |
| Vehicle OTA breaks helper start-up or Binder protocol | LOW | Detected via repeated readback failure. Banner in Settings: "Native control unavailable after update — investigating". Operator can defer the OTA. |

## 9. Open questions

1. **Helper bootstrap on first boot.** First-ever start needs RSA pairing on the
   ADB endpoint. Phase 1's existing read path already solves this; reuse the same
   pairing flow for write bootstrap. Confirm in 2b prototype.

2. **Helper port collision.** Pick a non-popular high port for the loopback socket
   (e.g. 8765) and document.

3. **Layer C action visibility on Leopard 3.** When a Layer C action's readback says
   "not supported" — keep silent in UI or show greyed-out? Decision at 2e when UI
   touches happen.
