# Phase 2 — Native write channel for comfort controls

**Status:** draft, awaiting brainstorming + approval
**Author:** generated 2026-05-27 after Phase 1a S9/S10/S5 validation
**Branch:** `feature/native-data-stack`
**Replaces:** existing D+ `sendCmd` path for vehicle comfort actions

---

## 1. Goal

Add a one-way **write channel** to native vehicle autoservice so BYDMate can control comfort
hardware (HVAC, seat heat/vent, mirror fold, sunroof, windows, locks, lights) without going
through the third-party D+ app. Read channel from Phase 1 stays as-is.

## 2. Why

D+ as control intermediary has four problems:

1. **Outside our trust boundary.** D+ is third-party, can OTA-change command surface anytime.
2. **Latency.** D+ proxy adds ~300-700 ms per command vs direct Binder ~10 ms.
3. **No state echo.** D+ does not surface confirmation of write success.
4. **Coverage gaps.** Some comfort fids have no D+ voice command (mirror fold being our
   recently discovered example — operator has hardware button but no D+ surface).

Native write removes all four.

## 3. Scope

### 3.1 In scope

Strictly **comfort layer** — operator-facing convenience hardware that has a steering-wheel
or panel button:

- HVAC: AC on/off, target temperature, fan level, mode (auto/manual), defrost front/rear,
  recirc, dual mode.
- Seats: heat level (0-5), vent level (0-5) — driver + passenger.
- Mirrors: fold/unfold (single button mapped to enum 1↔2).
- Sunroof: open / close / vent (enum 1/2/5 from S10).
- Sunshade: position 0-100% (validated S10).
- Windows: position 0-100% — driver / passenger / rear-left / rear-right.
- Locks: lock / unlock (enum 1↔2 from S10).
- Lights:
  - Inside ambient on/off.
  - Front fog on/off.
  - Rear fog on/off.
  - Hazard on/off (with user-approval gate — flashing lights = visible to others).

### 3.2 Out of scope (NEVER write)

- **Drive controls** — brake, throttle, gear, steering, EPB, ADAS configs.
- **Firmware** — anything writing to `/system`, BMS, MCU, CAN bus, gateway, OTA package.
- **Safety** — airbag arming, ABS thresholds, seatbelt enforcement, alarm state.
- **Powertrain** — drive mode (Eco/Normal/Sport), energy mode, e-pedal mode. UI surfaces
  it as "read-only telemetry" only.
- **Service intervals / odometer** — anything that could alter resale records.

This list is **enforced at allowlist layer** (see §6). Any write to a non-allowlisted
(fid, dev) pair is rejected before reaching native.

## 4. Architecture challenge

`transact=6` (setInt) on autoservice fails with `SecurityException` when invoked directly
from ADB WiFi shell (validated S2 attempt, see memory
`reference_autoservice_write_channel.md`). Autoservice checks the calling uid against an
internal allowlist; only first-party system processes pass.

### 4.1 Two paths considered

**Path A — DataStream bridge to shell-uid daemon.**
A long-running native helper started via `app_process --nice-name=...` runs under shell
uid (which IS in autoservice's allow-list). BYDMate connects to its TCP loopback via a
small binary protocol (writeByte opcode + 3× writeInt + readByte status + readInt
payload) and forwards write requests. Helper performs `Binder.transact(6, ...)` directly.

Pros: clean, fast (sub-50 ms round trip), survives BYDMate restart if daemon kept alive.
Cons: daemon needs initial activation via on-device ADB shell every cold boot
(self-respawn possible if seeded in user-script start-up).

**Path B — Each-call ADB-on-device shell pipe.**
BYDMate reuses its existing `AdbOnDeviceClient` (827 LOC already in main, used for read
fallback in older diagnostics) to push a single `service call autoservice 6 ...` command
each time.

Pros: zero daemon, easier first prototype.
Cons: ~300-500 ms per call (slower than D+), still needs initial ADB pairing, and breaks
if user toggles "Wireless ADB" off between sessions.

**Recommendation:** Path A. Path B as fallback only.

### 4.2 Daemon lifecycle (Path A)

- **Cold boot:** BYDMate's `BootReceiver` → `WorkManager` queues a one-shot `DaemonStartWorker`
  that does:
  1. `AdbOnDeviceClient` opens `127.0.0.1:5555`, RSA-pairs (one-time).
  2. Pushes `byd-comfort-daemon` binary to `/data/local/tmp/`.
  3. Runs `setsid app_process /data/local/tmp -Xnoimage-dex2oat /system/bin/--nice-name=byd-comfort-daemon com.bydmate.comfort.Daemon &`.
  4. Daemon listens on `127.0.0.1:<port>`.

- **Hot path:** BYDMate process opens TCP socket to daemon, sends `{opcode, dev, fid, value}`,
  reads `{status, readback}`.

- **Restart:** Daemon survives BYDMate process death but dies on Android reboot. BootReceiver
  re-seeds it. Heartbeat from BYDMate; on failure, re-spawn via ADB.

## 5. Safety / constraints

These are **hard rules**, enforced at allowlist + dispatch layers, not optional config.

1. **Speed gate.** If `Speed > 0` km/h (autoservice fid `-1807745016` dev=1013), block:
   windows down (already exists in current `ActionDispatcher` for D+ commands — preserve).
   Allow: AC, seats, lights (safe at any speed). Mirror fold: block above 5 km/h (folds
   only when parked anyway).

2. **Operator first-write gate.** First time the app writes ANY autoservice fid:
   modal "BYDMate will now control vehicle X. Allow? [Once / Always / Never]". Stored in
   per-fid map. Defaults to **off** until operator approves.

3. **Allowlist.** Hard-coded `WRITE_ALLOWLIST` map in code: `(fid, dev) → (value range,
   category)`. Anything outside the list — including category "drive" or "firmware" —
   is rejected pre-dispatch. New entries require code change + review.

4. **Audit log.** Every write call: timestamp, fid, dev, requested value, readback value,
   status. Stored in Room DB `vehicle_write_log` table. Retained 30 days. Surfaced in
   Settings → "Recent vehicle actions".

5. **Readback verification.** After every write, immediately `transact=5` to read the same
   fid (different dev if needed — see Phase 1 read/write namespace split in memory). If
   readback ≠ expected, mark action FAILED in log. UI shows error.

6. **Undo capability.** Maintain `last_known_state` per fid; "Undo" surfaced for 10 s
   after each comfort action. Tap → re-issues opposite write.

7. **Kill switch.** Settings → "Native vehicle control" toggle. Default ON post-onboarding.
   When OFF, all `setInt` writes return blocked status, UI falls back to D+ path (degraded
   mode for Smart Home, no UI breakage).

8. **No batch writes.** One fid per call. No multi-fid Binder transactions. Limits blast
   radius if a malformed call reaches autoservice.

## 6. Read/write fid namespace split

Phase 1 read fids are in `dev=1000` (HVAC namespace) for seats and AC.
Reference catalog suggests write fids live in `dev=1001` with **fid IDs offset by some
delta** and **values shifted by +1** (see memory `reference_autoservice_write_channel.md`).
Examples to verify on first probe:

| Param | Read (dev=1000) | Write (dev=1001?) | Value mapping |
|---|---|---|---|
| seat_heat_D | fid=702545948 → 0..5 | fid=1125122xxx? | value = read + 1 → 1..6 |
| seat_vent_D | fid=702545944 → 0..5 | fid=1125122xxx? | value = read + 1 |
| ac_temp_main | fid=1077936168 → °C | dev=1001 same fid? | direct °C |

First task in Phase 2b prototype: **write probe matrix** — try `setInt` against candidate
(fid, dev) pairs with a known harmless value (e.g. seat heat current level — write same
value as read), check readback, confirm "shift +1" hypothesis on a few entries before
generalizing.

## 7. Smart Home / Alice integration

Current path: Yandex Alice → VPS Express → BYDMate polling (2.5 s) → D+ `sendCmd`.

Phase 2 changes the **final hop only**: BYDMate polls VPS, on receiving a command
dispatches via native daemon if `WRITE_ALLOWLIST` covers it, else falls back to D+.
VPS API unchanged. Alice UI unchanged.

Latency win for Alice users: ~500 ms saved per command (D+ proxy removed).

Coverage: Smart Home already has 18 devices; most map cleanly to native fids. Mirror fold
(not in D+) becomes newly available to Alice voice commands.

## 9. Open questions (for brainstorming)

1. **Daemon binary distribution.** Bake `byd-comfort-daemon` into APK and push from app
   to `/data/local/tmp/` on first run? Or ship as separate split-APK? Single-APK keeps
   install simple; split-APK keeps daemon updatable independently. Lean: single-APK first.

2. **Daemon survival across DiLink ADB toggle.** If operator disables Wireless ADB on the
   head unit, daemon dies and can't be respawned. Show prominent banner "Native control
   disabled — turn Wireless ADB back on"? Or silent fallback to D+? Recommendation: banner
   + auto-fallback.

3. **Permission UI.** First-time write modal — show once per category (AC / seat / mirror /
   window / lock / light) or once per fid? Recommendation: per category (5 prompts max,
   not 20).

4. **Mirror fold:** worth adding as Smart Home device immediately (Alice "сложи зеркала")?
   This is one of the unlock-from-D+ wins.

5. **Sunroof:** binary fold control is easy (1/2/5); position control would need a
   `_pos` write fid that may not exist on Leopard 3 (need probe). Acceptable scope:
   ship 3-state control only (open / closed / vent) for v1.

6. **Lock/unlock from app.** Operator-sensitive — accidentally locking-out from outside
   is plausible. Should writes to locks require **biometric reauth** even after first-write
   approval? Recommendation: yes for `LockToState(2)` (locking), no for unlocking.

7. **Window position write granularity.** Operator panel button is press-and-hold. Do we
   want `setWindow(pos_percent)` (closed-loop) or `setWindow(direction)` (open-stop)?
   Closed-loop is nicer UX; need to verify daemon can poll position fast enough during
   move (~10 Hz).

8. **Audit log retention.** 30 days enough or do we want longer? Storage minimal but
   privacy consideration if vehicle changes owner.

## 10. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Daemon disabled by autoservice update (OTA) | HIGH | Detect early via readback failure; ship in-app banner + D+ fallback path stays available |
| Wrong (fid, dev, value) triggers undefined hardware state | CRITICAL | Allowlist + readback + manual probe matrix in 2b before generalizing |
| Privacy: audit log of operator actions leaks | LOW | Stored locally only; never sent to VPS or telemetry |
| First-write modal user fatigue | MEDIUM | Per-category not per-fid; default "Always" option |
| Daemon respawn loop on broken ADB pairing | LOW | Exponential backoff; max 3 retries before going dark with notification |
| Yandex Alice latency expectations | LOW | Even on fallback (D+) path it was 2 s; we improve, never degrade |
